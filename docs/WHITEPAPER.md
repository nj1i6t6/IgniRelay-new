## 0. 摘要

兩千多年前，烽火台用一束火光把「有事發生」傳過群山——不需要電力、不需要基地台，只需要一座座彼此看得見的塔。

**烽傳 IgniRelay 把同一個想法帶回現代：當行動網路消失，每支手機、每個低成本的中繼節點，就是一座烽火台。**

烽傳是一套「場域安全網」系統：營隊、賽事、工地、校外活動、登山路線、災區安置點——任何把一群人聚在一個範圍裡的場合，都是一個「場域」。場域裡的每個人用手機掃一個 QR code 加入，之後系統持續做三件事：

1. **被看見**——手機定期發出匿名的存在信標，管理者知道誰還在、最後出現在哪。
2. **能求救**——一鍵 SOS，自帶位置與安全狀態（受困／需醫療／已安全…），在所有可用的通道上擴散。
3. **留下最後足跡**——就算人失聯了，他走過的節點、遇過的手機，都替他留下了加密簽章的足跡。

它有兩種形態：**有網路的場域**直接走網際網路上雲端服務，場域主在任何地方用瀏覽器看全場；**完全沒有網路的場域**靠手機藍牙 mesh 與實體 LoRa 節點接力，管理者在現場的本地後台看全場——同一個 App、同一個 QR、同一種加密事件信封、同一套後台。

在求救之後，烽傳接上了合作專案 **E-CARE**（校內團隊開發的 AI 緊急應答系統）：求救者可以進入 AI 對話獲得安撫、風險評估與急救引導；求救案件自動進入 E-CARE 的案件儀表板。E-CARE 的 AI 完全在本地硬體上推論（自行微調的 LLM），與烽傳「不依賴外部基礎設施」的哲學一脈相承。

目前狀態：通訊協議已凍結並有三平台一致性測試（✅）、App 核心施工中（🔧）、節點／後台／雲端／AI 串接規格已全數凍結並排程（📋）。

## 1. 問題：當網路消失的時候

### 1.1 三個殘酷事實

1. **行動網路是災害中最先倒下的基礎設施之一。** 基地台依賴電力與回程網路；大型地震、風災、火災中，斷電與光纜中斷會讓整個區域瞬間失聯——而那正是人們最需要求救的時刻。
2. **很多地方本來就沒有訊號。** 山域步道、礦場、隧道工地、離島、偏鄉農場——這些場域的日常就是離線的，意外發生時沒有任何系統知道「他最後在哪」。
3. **就算網路還在，人也可能不在系統裡。** 營隊兩百個學員、賽事三千個跑者、工地百名工人——管理者靠的往往是紙本名冊與對講機，沒有任何即時的「誰在哪、誰出事了」。

### 1.2 三個場景

- **山難**：登山者在無訊號路段失聯。沒有最後位置，搜救範圍以平方公里計；每晚一小時，存活率就往下掉。
- **災區安置**：避難所裡誰需要醫療、誰已經撤離、誰失聯了——管理者用白板和喊話在管理數百人。
- **日常場域**：營隊點名、工地巡檢、賽事關門時間——大量人力花在「確認人還在」這件事上，而真正的緊急狀況反而最慢被發現。

### 1.3 現有方案的缺口

| 方案 | 問題 |
|---|---|
| 衛星通訊器（inReach 等） | 昂貴（設備＋月費）、人手一台不現實、單向為主、無場域管理面 |
| 無線電對講機 | 只有語音、無身分、無紀錄、無位置、需要人守著聽 |
| 消費級 mesh 聊天 App | 純靠手機密度、無管理後台、無硬體接力、訊息無結構（不能變成看板） |
| 商用人員定位系統 | 封閉、昂貴、需要預先佈建基礎設施與後端，災時不可得 |

缺的是一套**便宜、開放、離線也成立、且有管理面**的場域安全系統——這就是烽傳。

## 2. 解決方案總覽

### 2.1 一句話

**讓每個帶手機的人「被看見、能求救、留下最後足跡」——有網路時是雲端 SaaS，沒網路時是自成一格的 mesh 網路；「場域」本身就是可販售的服務單位，硬體是場域的離線增強包。**

### 2.2 兩種產品形態

| | 純軟體形態 | 軟硬整合形態 |
|---|---|---|
| 適用場域 | 營區、賽事、工地、校外活動（有行動網路） | 登山步道、礦場、災區、巡檢路線（無網路） |
| 加入方式 | 掃場域 QR | 掃場域 QR（**同一個**） |
| 傳輸 | 手機 → HTTPS → 雲端 | 手機 → BLE → 節點 → LoRa → 閘道 |
| 管理面 | 雲端後台（任何地方的瀏覽器） | 現場本地後台（同 LAN 瀏覽器，零外網依賴） |
| 升級路徑 | 加買節點硬體即升級為軟硬整合 | 閘道有回程網路時自動同步上雲 |

兩種形態＝同一個 App、同一個 QR 加入流程、同一種事件信封、同一套後台設計語言。客戶從軟體形態升級到軟硬整合，**什麼都不用換**。

### 2.3 走一遍

**場景 A——有網路的營隊（純軟體形態）📋**
場域主在雲端開一個場域，印出 QR 海報貼在報到處。學員掃碼加入，手機開始定期發匿名存在信標。場域主的儀表板上看得到全場人數與每個人的最後可信位置。一名學員夜間迷路按下 SOS——位置與安全狀態立刻出現在儀表板的 SOS 看板（永遠置頂、任何設定都遮蔽不了它），同時學員的手機跳出「需要進一步協助嗎？」——他進入 AI 對話，E-CARE 安撫他、確認他的狀態、給出等待救援時的指引。工作人員依方位前往，學員回報「已安全」，案件關閉，全程留有紀錄。

**場景 B——無網路的步道（軟硬整合形態）📋**
步道沿線每隔一段掛一個太陽能 LoRa 節點。登山者經過節點時，手機的存在信標被節點「撿走」，經 LoRa 一跳一跳接力回登山口的閘道。閘道的本地後台上，留守人員看得到每個人最後經過哪個節點。一名山友跌落邊坡按下 SOS——70 bytes 的求救幀數秒內跳回閘道，後台亮起。就算他所在位置沒有節點覆蓋，他的 SOS 也存在每一支路過山友的手機裡（data mule），走回有覆蓋的地方就自動轉發。閘道若接上衛星或行動網路回程，事件同步上雲，留守人員不用守在山下。

### 2.4 降級階梯（核心設計哲學）

烽傳不是「一條通訊路徑」，是一座**每層失效都有下一層接住**的階梯：

| 層 | 通道 | 失效時 |
|---|---|---|
| L1 | 網際網路（HTTPS 直上雲端） | ↓ 退到 L2 |
| L2 | 實體節點（BLE → LoRa → 閘道） | ↓ 退到 L3 |
| L3 | 手機 ↔ 手機 BLE mesh（相遇即同步） | ↓ 退到 L4 |
| L4 | Data mule（人的物理移動攜帶事件） | 最壞情況：**一個走過的人就是你的網路** |

每一層攜帶的是**同一份加密簽章的事件位元組**——這是整個系統最重要的一個設計決策（見 §3.1）。

## 3. 系統架構

### 3.1 一種信封走天下 ✅

系統的原子單位是 **EventEnvelope v3**：一個 141-byte 的 canonical 編碼信封，內含：

- **Ed25519 作者簽章**——誰發的、內容沒被改過，數學上可證。
- **field_id（簽進 canonical）＋ field_mac**——HMAC-SHA256 場域成員證明（截 16 bytes）：只有持有場域金鑰的成員能產生有效信封，外人塞不進來。
- **事件本體**——類型、匿名 ID、HLC 時間戳、payload。

關鍵性質：**傳輸層永遠不需要懂內容。** 同一份位元組原樣走 BLE GATT、走 LoRa（緊湊轉譯框架）、走 HTTPS POST——每個收到它的端點（手機、節點、閘道、雲端）都用**同一套驗證管線**重新驗證。這個決策的回報已經被驗證過一次：v1.2 計畫加入整個雲端服務時，**通訊協議一個位元組都不用改**。

### 3.2 組件與資料流

```
離線形態（軟硬整合）：

[手機 App] --BLE GATT（EventEnvelope v3，Ed25519+field_mac）--> [Field Node ×N]
     |  ^                                                          |
     |  └── 手機↔手機 BLE mesh（fallback、data mule）                | LoRa（緊湊框架，
     v                                                             |  HMAC+CRC，58–88B）
[手機 App（其他人）]                                          [Gateway（PC／樹莓派）]
                                                                   |
                                                       SQLite ＋ 本地 Web 後台
                                                                   |
                                                     [管理者瀏覽器（同 LAN，零外網）]

連網形態（純軟體）＋雲端：

[手機 App] ----HTTPS（信封原樣 b64 批次）----> [雲端場域服務（VPS，多場域）]
[Gateway]  ----批次同步（有回程網路時）--------^         |
                                              [場域主／工作人員瀏覽器：雲端後台]
                                                         |
                              SOS 通報轉發（HTTPS，經代理）v
                                              [E-CARE AI 應急後端（合作專案）]
```

| 組件 | 角色 | 狀態 |
|---|---|---|
| 手機 App（Flutter，Android 優先） | 發 PRESENCE／SOS／HAZARD／CHECKPOINT，收場域廣播；mapless 位置卡＋雷達視圖；SOS 後援對話 | 🔧 核心接線中 |
| Field Node（nRF54L15＋SX1262） | BLE 收手機事件→驗章驗 MAC→去重→優先佇列→LoRa store-and-forward | 📋（先模擬器後實機） |
| Gateway（Python，PC→樹莓派） | LoRa 彙整、驗證、SQLite 存檔、本地 Web 後台、CSV/JSON 匯出 | 📋 |
| 雲端場域服務（Owner VPS） | 多場域、三角色、可見性政策、ingest、雲端後台、自訂地圖 | 📋（規格已凍結） |
| E-CARE 串接 | SOS 後援 AI 對話＋案件通報（見 §4） | 📋（規格已凍結） |

### 3.3 事件詞彙

| 事件 | 內容 | 設計重點 |
|---|---|---|
| PRESENCE | 匿名 ID＋電量＋位置證據 | 週期信標；「被看見」的最小單位 |
| SOS（STATUS_UPDATE） | 安全狀態（受困／需醫療／已安全…）＋自帶位置 | 最高優先插隊；**取消＝後送「已安全」狀態覆寫**，不增加新訊息型別 |
| HAZARD | 危險類型＋嚴重度＋位置＋描述 | 落石、火點、坍方等場域危險回報 |
| CHECKPOINT | 經過節點記錄 | 「最後足跡」的骨幹 |
| ADMIN_BROADCAST | 場域管理廣播 | 撤離指令、集合通知 |
| NODE_RECEIPT | 節點回執 | 手機顯示「已送達節點」——求救者知道訊息出去了 |

### 3.4 實體節點與 LoRa 接力 📋

- 硬體：Nordic **nRF54L15**（BLE）＋ Semtech **SX1262**（LoRa），太陽能供電目標；台灣 AS923（923MHz）頻段，功率與占空比依法規。
- LORA-WIRE 緊湊框架：每幀 = 標頭 11B＋本體＋HMAC 8B＋CRC16 2B。實測預算：PRESENCE 58B、SOS 70B、HAZARD ≤88B——在 LoRa 的嚴苛空中時間限制下仍能多跳接力。
- Store-and-forward：節點斷鏈時排隊保存，鏈路恢復即補傳；事件 ID 全網冪等，重送永遠安全。
- 工程路線：**先用模擬器把整條鏈跑到全綠（含混沌測試），通過採購關卡才買硬體**——不把資金押在未驗證的設計上。

### 3.5 一致性與去重 ✅

- 每事件全域唯一 event_id，全網冪等（任何端點重複收到只算一次）。
- **HLC（混合邏輯時鐘）**＋重放窗口：不依賴任何端點的時鐘正確，仍能排序與防重放。
- 手機相遇時的**集合調和同步**（Bloom filter＋IBLT）：兩支手機快速找出彼此缺的事件差集，短暫交會也能完成補同步——這就是 data mule 成立的技術基礎。

### 3.6 工程方法：可驗證的誠實

這個專案對「做完了」的定義是苛刻的，幾個機制值得一提：

- **凍結契約**：信封規格、LoRa 框架、API 規格逐一凍結版本，跨端變更必須同步三平台。
- **一致性語料（conformance corpus）**：由單一生成器產出 217 個跨端測試樣本（信封 104、分塊 20、IBLT 52、Bloom 30、負樣本 11），Dart／Kotlin／Swift 三端用**同一份**語料鎖住位元組級一致性；三端 13 個協議常數有自動 parity 檢查。
- **架構防腐**：UI／應用／通訊／原生四層的 import 邊界由 lint 工具強制（違規即建置失敗）。
- **測試基線**：App 目前 469 個測試全綠（✅，2026-06-10 基線）。

## 4. 智慧應急層——與 E-CARE 的雙專案架構

> **定位一句話：烽傳是神經系統，E-CARE 是大腦。**
> 烽傳負責「訊息在沒有網路的世界裡流動」；E-CARE 負責「聽懂求救的人，並接住他」。
> 兩個專案各自完整、各自可獨立運作，在「SOS 之後的那一刻」接合。

### 4.1 E-CARE 是什麼

**E-CARE** 是校內合作團隊開發的 AI 緊急事件輔助系統（[github.com/rungyu0721/Ecare](https://github.com/rungyu0721/Ecare)；團隊成員名單〔待填〕）：FastAPI 後端＋Flutter 前端，能聽（語音辨識）、能懂（事件分類與語意理解）、能判斷（風險評估）、能安撫（LLM 對話）、能引導（急救知識）、能說（語音合成）——而且**整套 AI 在本地硬體上推論**，不依賴任何外部雲端 AI 服務。

它處理六大類事件：醫療急症、交通事故、火災、暴力事件、可疑人士、噪音；風險分級 Low／Medium／High，高風險自動升級為通報案件。

### 4.2 E-CARE 的 AI 技術棧（本節成果歸屬 E-CARE 團隊）

**① 自行微調的本地 LLM。** 以 Qwen2.5 為基底，用自製繁體中文緊急對話資料集微調出專用模型 `ecare-v4`（完整的訓練資料生成→LoRA 微調→GGUF 量化流程都在其 repo 內），經 Ollama 在本機 GPU 推論。低溫採樣（temperature 0.1）＋強制結構化 JSON 輸出——回覆、風險分數、追問、語意分析一次產出，前端可程式化消費。**推論不出機房**：這對緊急應答系統是雙重正確——隱私（求救內容不經第三方）與韌性（外網斷了 AI 還在）。

**② 心理急救（PFA）原則內建於對話策略。** 系統提示詞將 Psychological First Aid 原則寫成硬規則：先接住情緒再進入協助、一次只追問一個最重要的問題、不逼問創傷細節、不用「保持冷靜」這類無效安撫（改用「先慢慢來，我會一步一步陪你整理」）、**明確禁止 AI 假裝已通知警消**（只能協助整理資訊與建議撥打 110/119）——這是有專業依據的對話設計，也是對 AI 幻覺的防護。

**③ 通報者角色感知。** 區分「本人受害／旁觀者／照顧者家屬／代他人通報」並切換話術：對本人先確認「你現在安全嗎」，對旁觀者提醒保持距離不要介入，對照顧者同時支持照顧者與傷病者。

**④ 語音情緒辨識。** 從錄音中抽取聲學特徵（MFCC 及其差分、梅爾頻譜、過零率、頻譜質心、能量），以自行訓練的分類器辨識 panic／fearful 等情緒並給出信心分數——**情緒分數直接加權風險評估**：聽起來在恐慌的「我沒事」，系統不會當作沒事。

**⑤ 災難領域調校的語音辨識。** Whisper 加上緊急領域提示詞（持刀、家暴、失火、沒呼吸……），再疊一層領域錯字修復：同音誤轉如「拿到刀」→「拿刀」在威脅語境下自動修正——一般 ASR 在最關鍵的詞上最容易錯，這層修復直接救回風險判斷。

**⑥ 雙層風險引擎：規則底線＋LLM 同理。** 規則層（關鍵字庫＋語境正規表達式＋否定句／不確定句語意處理）做**確定性的安全底線**：就算 LLM 判斷失誤，「有人拿刀追我」這類訊號會被規則層強制拉到高風險。LLM 在底線之上負責同理、追問與整理。語意層有 36 筆語意回歸＋52 筆多輪上下文回歸測試鎖住行為（含「沒受傷」否定句、「警察已經到了」事後回報等易錯情境）。

**⑦ 本地語音合成。** CosyVoice2-0.5B 本地 TTS：高風險時自動播報行動導向短句（「系統已列為高風險通報，請確認患者是否有正常呼吸」），並做了**預合成快取**（對話回覆產生的同時就開始合成語音）壓低播報延遲；本地模型失效時退到系統內建 TTS。

**⑧ 資料化的急救知識庫＋使用者情境圖譜。** CPR／AED、成人與嬰兒異物哽塞、大量出血（含異物插入）、燒燙傷、癲癇、中暑、胸痛、骨折（含車禍）、中風等情境的引導語資料化管理；Neo4j 圖資料庫記錄使用者情境脈絡供多輪對話參照。後端有 77 個單元測試與一鍵回歸檢查流程。

### 4.3 整合架構 📋（規格已凍結，v1.3）

**App 端——SOS 後援對話：**

```
按下 SOS
  ├─► SOS 照常發出（mesh＋雲端）……這條不等任何人、不依賴 E-CARE
  └─► 彈出詢問：「需要進一步協助嗎？」
        └─ 同意 → 對話介面
              ├─ 無網路 → 輸入文字觸發關鍵字 → 本地急救圖卡
              │   （內容取自 E-CARE 急救知識庫快照，打包在 App 內，完全離線可用）
              └─ 有網路 → E-CARE AI 對話
                  （安撫＋風險評估＋追問；AI 的判斷只作顯示，不改寫求救狀態）
```

**管理端——案件自動通報：** SOS 事件由雲端服務自動轉發為 E-CARE 案件（含座標與對帳標記），出現在 E-CARE 的案件儀表板；求救者後續的狀態變化（已安全／救援已抵達…）同步更新案件狀態歷程。烽傳自己的雲端後台則有一個「E-CARE 通報」分頁，直接呈現 E-CARE 的案件與 AI 風險分析——**資料與智能來自 E-CARE，管理動線整合在烽傳後台**。

**AI 援助的降級階梯：**

| 層 | 條件 | 使用者得到什麼 | 狀態 |
|---|---|---|---|
| L1 | 手機有網路 | 完整 LLM 對話（安撫、追問、風險評估、急救引導） | 📋 |
| L2 | 斷網 | 本地關鍵字急救圖卡（E-CARE 知識庫離線快照） | 📋 |
| L3 | 純 mesh | SOS 結構化欄位照常流動；閘道有網路時案件照常通報 E-CARE | 📋 |
| L4 | mesh 中繼 AI 問答 | 經節點接力的短訊息問答 | 💡 roadmap |

### 4.4 整合的設計紅線

1. **SOS 零延遲**：一切 AI 功能都發生在 SOS 已送出之後；E-CARE 連不上、回應慢、整個服務下線——SOS 的發送與擴散**完全不受影響**（以架構層級的隔離強制，非僅約定）。
2. **隱私邊界**：傳給 E-CARE 的只有事件欄位（類型／座標／狀態／匿名 ID），**不含任何真名電話等個資**；E-CARE 的存取一律經烽傳的代理層與金鑰管控。
3. **零改動掛接**：整合不要求 E-CARE 修改任何一行程式碼——黏合層全部在烽傳側。兩個專案保持各自獨立的開發節奏與完整性。
4. **可用性誠實**：E-CARE 跑在學校資源上，「暫時不可用」被當作正常狀態設計——App 無聲降級回圖卡模式，永不轉圈卡死。

### 4.5 為什麼這個合作在技術上是對的

- **同一個哲學**：E-CARE 的 AI 不出機房、烽傳的通訊不靠基地台——兩個專案在「不依賴外部基礎設施」上完全同調，合起來是一條從求救、傳遞到應答都離線可運作的完整鏈路。
- **窄介面**：整合面只有三個 HTTP 端點（對話、建案、狀態更新），兩專案可以各自獨立演進與展示。
- **互補而非重疊**：烽傳沒有 AI、E-CARE 沒有離線通訊——拼起來剛好是一張完整的拼圖。

## 5. 沒有地圖的地方怎麼指路

### 5.1 誠實定位哲學 📋

災防系統最危險的事情之一，是讓搜救者以為「點在地圖上的位置」就是「人現在的位置」。烽傳從文案到資料模型都貫徹一個原則：**這是「最後可信位置」，不是「目前位置」。**

每筆位置都帶來源與年齡，並按年齡自動降級可信度：≤2 分鐘＝高（實心顯示）、≤10 分鐘＝中、更舊＝低（明確標示「N 分鐘前」）。位置可以來自本人 GPS，也可以來自「誰的手機在哪裡聽到過他」的鄰近證據——系統融合多種證據給出估計與不確定半徑，**且推導值永不寫回傳輸層**（顯示層的智慧不污染資料的真實）。

### 5.2 雷達相對方位視圖 📋

沒有地圖時，搜救者真正需要的是：「他在我的哪個方向、多遠、資訊多舊」。烽傳提供北朝上的雷達視圖：以自己為圓心，場域成員以方位角＋距離畫成點，距離環自動縮放，點的顏色亮度反映可信度年齡，SOS 點永遠以警示色置頂。純幾何計算、零地圖依賴、零網路依賴。

### 5.3 場域自訂地圖 📋

線上地圖（Google／OSM 圖磚）在烽傳的世界裡有三個致命傷：斷網拿不到、授權有疆界、災區圖資過時。烽傳的答案是**讓場域主自己上傳地圖**：

- 一張營區手繪圖、空拍圖、疏散平面圖都可以。
- 場域主在圖上釘 2–10 個對位點（每點填入實際經緯度），系統以最小平方法擬合相似變換（旋轉＋縮放＋平移，數學上禁止鏡射），**即時顯示擬合殘差（公尺）**——場域主當場知道自己的圖準不準。
- 此後全場人員、節點、SOS 即時疊加在這張圖上；手機 App 與管理後台用同一組對位點各自重算變換，並以同一份跨端驗證向量鎖住兩端一致（誤差容忍 0.01 像素）。

**自己的場域，自己的圖**——比任何官方圖磚都貼近現場，而且零外部依賴、零圖資授權問題。

## 6. 安全與隱私

### 6.1 信任邊界總覽

| 邊界 | 機制 |
|---|---|
| 手機 ↔ 手機／節點（BLE） | EventEnvelope v3：Ed25519 作者簽章＋場域 HMAC（field_mac），141B canonical ✅ |
| 節點 ↔ 節點 ↔ 閘道（LoRa） | 緊湊框架：場域預過濾＋HMAC-SHA256（金鑰域分隔派生）＋CRC＋去重＋重放窗 📋 |
| 閘道 ↔ 瀏覽器（LAN） | Token 驗證、僅綁 LAN、零外網資源 📋 |
| 手機／閘道 ↔ 雲端（HTTPS） | TLS **加上**信封自證（雲端用與現場閘道同一套驗證管線重驗每一封）📋 |
| 雲端 ↔ E-CARE（跨專案） | 一律經代理＋金鑰；只送事件欄位、零個資 📋 |

### 6.2 逐跳驗證：不信任何中繼

事件每經過一個端點（手機、節點、閘道、雲端）都被**重新完整驗證**——簽章、場域 MAC、去重、重放窗。一條鐵則貫穿全系統：**「外層有 TLS」不能取代信封驗證**。被攻陷的節點、偽造的上傳者、重放的舊封包，在每一跳都會被擋下；data mule 模式（陌生手機代傳他人事件）也因此天然安全——代傳者改不了內容，也偽造不了來源。

### 6.3 匿名設計

無線電上**永遠沒有真名**。每個成員以隨機產生的匿名 ID（anon8）存在，與簽章金鑰分離保管。「3 號帳篷的王同學」這類對應，只存在場域主後台的備註欄（服務層資料），不上傳輸層；場域結束，對應即可銷毀。

### 6.4 SOS 永不遮蔽

場域主可以設定成員之間是否互見足跡（隱私政策），但**任何政策都遮蔽不了 SOS**——求救永遠對全場域可見。這是寫進規格的鐵則，不是預設值。

### 6.5 金鑰治理

每個場域有獨立金鑰（場域即信任域）：單一場域金鑰外洩，爆炸半徑只有那一個場域，且有換鑰程序。雲端伺服器的角色是「持有場域金鑰的雲端閘道」——由於場域金鑰本就由全場成員共持，多一台加固的伺服器並不實質改變威脅模型，卻換來雲地兩端共用同一套驗證程式碼（少一套實作＝少一片攻擊面）。

### 6.6 已知限制（誠實聲明）

- 手機↔手機 mesh 依賴手機密度——這正是實體節點存在的理由。
- 離線形態下，成員間可見性政策由 App 端遵守（場域內事件本就全員接力）；伺服器級的強制只在連網形態成立——我們在文件與行銷上如實描述此差異。
- 場域金鑰由全場成員共持（QR 即金鑰載體），防的是「場外人」而非「惡意成員」；成員級的細粒度權限是 💡 roadmap。

## 7. 商業模式：「場域」即服務

### 7.1 服務單位

賣的不是 App（App 免費），是「**場域**」：一個有時限、有範圍、有管理者的安全網實例。場域主開通場域→印 QR→成員掃碼即入→場域結束即歸檔。一所學校的一次校外教學、一場馬拉松、一個工地的一季施工，各是一個場域。

### 7.2 客群與形態

| 位置可信度 | ≤2 分鐘＝高／≤10 分鐘＝中／更舊＝低；不確定半徑隨估計輸出 |
